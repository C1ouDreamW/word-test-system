# 分支管理与提交规范

---

## 一、仓库信息

- 平台：GitHub
- 主分支：`main`
- 开发分支：`dev-a`（A） / `dev-b`（B）

---

## 二、分支策略

```
main ─────────────────────────●───●───●──── 稳定版本
         ↘                    ↗   ↗   ↗
dev-a      ●──●──●──●────────  A 日常开发
         ↘                    ↗
dev-b      ●──●──●──●────────  B 日常开发
```

| 分支 | 作用 | 谁用 |
|------|------|:--:|
| `main` | 可运行的最新稳定版本 | 两人 |
| `dev-a` | A 的开发分支 | A |
| `dev-b` | B 的开发分支 | B |

**规则：**
- 每天收工时，各自把当天代码合并到 `main`
- 每天开工前，从 `main` 拉最新代码到自己的分支：`git checkout dev-a && git merge main`
- **禁止直接在 main 上改代码**

---

## 三、操作流程

```bash
# === 每天开工（拉最新代码） ===
git checkout main
git pull origin main
git checkout dev-a            # 换成自己的分支
git merge main                # 把 main 合进来

# === 开发中 ===
git add .
git commit -m "feat(考试): 完成判卷接口"

# === 每天收工（推到远程） ===
git checkout main
git pull origin main          # 先拉别人的
git merge dev-a               # 合并自己的
# 解决冲突（如果有）
git push origin main

git checkout dev-a
git merge main                # 同步自己的分支
```

## 四、Commit 格式

```
type(scope): 简要说明
```

### type — 提交类型（必填）

| type | 含义 |
|------|------|
| `feat` | 新功能 |
| `fix` | 修复 bug |
| `docs` | 文档变更 |
| `refactor` | 重构（不改变功能） |
| `chore` | 构建/工具/依赖变更 |
| `style` | 代码格式（不影响逻辑） |

### scope — 模块范围（必填）

| scope | 含义 |
|------|------|
| `项目` | pom.xml、application.properties、启动类 |
| `数据库` | SQL 脚本、实体类 |
| `单词` | 单词 CRUD |
| `考试` | 考试管理、组卷、判卷 |
| `统计` | 成绩查询、统计 |
| `字符云` | 字符云数据、前端渲染 |
| `前端` | HTML / CSS / JS 页面 |
| `登录` | 用户登录 |

### 示例

```bash
feat(项目): 初始化 Spring Boot 项目
feat(数据库): 添加建表 SQL 和预置数据
feat(单词): 完成 WordDao 和 WordService
feat(考试): 实现组卷算法和随机打乱
feat(前端): 完成答题页面和倒计时
fix(考试): 修复判卷忽略大小写的问题
fix(前端): 修复倒计时归零后重复提交
refactor(考试): 抽取判卷公共方法
chore(项目): 添加阿里巴巴规约插件
docs(项目): 更新 README
```

## 五、禁止事项

| ❌ 禁止 | ✅ 应该 |
|---------|--------|
| 直接 push 到 main | 先合到自己的分支，再 merge 到 main |
| 最后一天堆一堆提交 | 每天至少提交 1 次 |
| commit 写 "修复" "更新" | 写清楚改了哪个模块 |
| 不 pull 直接 merge | 先 pull，解决冲突再 push |
| 把自己分支的 jar/class 文件提交 | 加到 .gitignore |
